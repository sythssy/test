# 项目核心规则（织梦AI小说）

## 1. 项目概述 (Project Overview)

我们将开发一个名为“织梦ai小说”的专业 AI 辅助小说创作 SaaS 平台。

- 前端框架: Next.js 14 (App Router) + TypeScript + Tailwind CSS
- UI 组件库: Shadcn UI + Lucide Icons
- 富文本编辑器: Tiptap（无边框沉浸式，支持划线弹出悬浮栏）
- 数据库与鉴权: Supabase (PostgreSQL + Supabase Auth)
- AI 底层引擎: Dify API（按需接入多模型，通过不同的 API Key 路由）
- 核心商业模式: 按字数实时扣费。用户算力以 `word_balance`（字数余额）结算
- 提示词管理策略: 前后端分离管理 AI 提示词，运行时动态读取数据库配置

## 2. 数据库设计 (Database Schema - Supabase)

必须严格按照以下结构建立表与关联。

### 表 1: `users`（用户核心表）

- `id`: uuid (Primary Key, 关联 Supabase Auth)
- `email`: string
- `word_balance`: integer（默认 50000，字数余额）
- `role`: string（默认 `user`，管理员为 `admin`）
- `status`: string（默认 `active`，可选：`active`、`banned`）

### 表 2: `books`（小说工程表）

- `id`: uuid
- `user_id`: uuid (Foreign Key -> users.id)
- `title`: string
- `cover_url`: string
- `current_conversation_id`: string（关键字段，保存该书右侧多轮对话 Dify 会话 ID）
- `created_at`: timestamp

### 表 3: `chapters`（章节表 - 构成左侧目录）

- `id`: uuid
- `book_id`: uuid (Foreign Key -> books.id)
- `title`: string
- `content`: jsonb（Tiptap 数据格式）
- `word_count`: integer
- `order_index`: integer（排序用）

### 表 4: `ai_prompts`（后台动态提示词表）

- `id`: uuid
- `action_type`: string（唯一键，枚举：`rewrite`、`anti_ai`、`outline`、`chat` 等）
- `name`: string（中文展示名）
- `system_prompt`: text（发送给 Dify 的系统指令）
- `dify_api_key`: string（对应 Dify App Key，用于切换模型）
- `is_active`: boolean（默认 true）

### 表 5: `billing_logs`（账单流水表）

- `id`: uuid
- `user_id`: uuid
- `action_type`: string
- `cost_words`: integer（本次消耗 = 输入字数 + 输出字数）
- `created_at`: timestamp

### 数据与索引补充约束

- 所有业务查询必须带 `user_id` 或与 `book_id -> user_id` 关联校验，禁止越权读取。
- `chapters(book_id, order_index)` 建联合索引，保证章节列表排序与切换性能。
- `billing_logs(user_id, created_at)` 建索引，支持账单倒序查询。
- `ai_prompts(action_type)` 建唯一索引，保证动作类型唯一映射。
- `users(role)` 与 `users(status)` 建索引，支持权限判断与封禁拦截。

- 页面状态：loading / empty / save-success / save-error 均需定义。

## 4. 后端 API 与 AI 流式扣费逻辑（核心骨架）

必须在 Next.js 服务端路由建立以下两个严格隔离的接口。

### 接口 A: `/api/generator`（无状态一次性操作）

触发场景：扩写、去痕、润色、划线修改。

执行逻辑：

1. 接收前端传入 `action_type` 与 `user_text`。
2. 计费拦截：`input_words = user_text.length`，余额不足则立即返回错误。
3. 动态读取：查询 `ai_prompts` 获取对应 `system_prompt` 与 `dify_api_key`。
4. 请求 Dify Workflow API，注入上述变量。
5. 流式返回与扣费：
   - 流中 token 透传前端
   - 结束时统计 `output_words`
   - 执行扣费 SQL: `word_balance = word_balance - (input_words + output_words)`
   - 写入 `billing_logs`
6. 不产生、也不保存任何 `conversation_id`。

### 接口 B: `/api/chat`（有状态多轮推演）

触发场景：右下角悬浮球对话。

执行逻辑：

1. 接收历史消息与 `books.current_conversation_id`（首次为空）。
2. 动态读取 `action_type = 'chat'` 的后台配置。
3. 请求 Dify Chat API。
4. 关键逻辑：从 Dify 流中捕获 `conversation_id`，立即更新到当前 `books`。
5. 流结束后统计提问字数 + 回复字数，扣除用户余额并写账单。

### 4.1 流式事件协议（新增，前后端强约束）

统一 SSE/流事件字段（最少）：

- `type`: `token` | `usage` | `conversation` | `done` | `error`
- `token`: 增量文本（`type=token`）
- `usage`: `{ input_words, output_words, total_words }`（`type=usage`）
- `conversation_id`: string（`type=conversation`）
- `error_code`: 见错误码标准（`type=error`）
- `message`: string（人类可读提示）

### 4.2 错误码标准（新增）

- `INSUFFICIENT_BALANCE`
- `UNAUTHORIZED`
- `FORBIDDEN`
- `PROMPT_NOT_FOUND`
- `PROMPT_INACTIVE`
- `DIFY_TIMEOUT`
- `DIFY_BAD_RESPONSE`
- `RATE_LIMITED`
- `INTERNAL_ERROR`

前端错误映射必须固定：

- 余额不足 -> 弹窗提示并阻止继续生成
- 未登录/无权限 -> 跳转登录或无权限页
- 超时/限流 -> Toast + 提供重试
- 配置缺失 -> Toast（提示联系管理员）

### 4.3 账务一致性要求（新增）

- 扣费与写流水必须在同一事务中提交，避免余额与账单不一致。
- 扣费前再次验证余额，防止并发透支。
- 前端显示余额以服务端最新返回为准，不做本地假扣费持久化。

## 8. 写作台 UI 与排版规范（高优先级新增）

本章节用于约束视觉与编辑体验，避免只完成功能但缺乏写作产品质感。

### 8.1 三栏布局与空间规范

- 左侧章节栏：固定宽度 260px，支持折叠；折叠后保留章节切换入口。
- 中间编辑区：最小宽度 760px；大屏内容区居中，正文可读宽度建议 720px~820px。
- 右侧聊天区：默认悬浮，展开后宽度 360px；不遮挡核心编辑光标区域。
- 顶部吸顶栏：展示作品名、章节名、保存状态、模型选择、AI 工具按钮。

### 8.2 视觉风格与组件状态

- 统一圆角、边框、阴影层级：卡片/弹窗/悬浮菜单保持一致视觉系统。
- AI 动作按钮使用主色系强调；删除、清空、覆盖等操作统一 danger 样式。
- 所有交互按钮必须定义 4 种状态：default、hover、disabled、loading。
- 弹窗统一采用 `Dialog`，二次确认文案需写明影响范围（如“将删除当前章节”）。

### 8.3 字体与排版设置（用户可调）

- 字体大小：提供 `14` / `16` / `18` / `20` 四档。
- 字体家族：至少提供“中文阅读字体”和“等宽字体”两类选项。
- 行高：`1.6` / `1.8` / `2.0` 可选。
- 段落缩进开关：支持首行缩进 2 字符（适配网文写作习惯）。
- 夜间模式：提供浅色/深色两套阅读主题，保证对比度。

### 8.4 编辑工具栏（Tiptap）功能清单

- 文本样式：加粗、斜体、下划线、删除线、字体标红。
- 结构样式：H1/H2/H3、引用、无序/有序列表、分割线。
- 快捷操作：撤销、重做、清除格式。
- 沉浸模式：开启后隐藏左栏和非核心按钮，仅保留正文与必要状态信息。

## 9. 编辑器增强功能清单（查重 + 局部 AI）

### 9.1 重复文字查找

- 作用范围：默认当前章节，可选“仅选中文本范围”。
- 检测模式：
  - 重复词（连续或高频短词）
  - 重复句（句子级相似重复）
- 交互流程：
  1. 点击“查重”
  2. 右侧出现结果列表（重复内容、出现次数、位置）
  3. 点击结果自动定位并高亮文本
  4. 支持“逐条处理”与“一键忽略”

### 9.2 文字局部润色与扩写

- 入口：
  - 选中文本后的 Inline Menu
  - 顶部 AI 工具栏（无选区时默认作用当前段落）
- 动作类型：
  - `润色`（提高清晰度与文采）
  - `扩写`（扩展细节与描写）
  - `精简`（压缩冗余表达）
  - `改写语气`（严肃/轻松/诗性等）
- 结果回写：
  - 先展示 diff 预览（原文 vs 生成文）
  - 用户点击“确认替换”后才写回正文
  - 支持“仅插入不替换”

### 9.3 自动保存与版本恢复

- 防抖自动保存：800ms~1500ms。
- 保存状态栏：`保存中` / `已保存` / `保存失败（可重试）`。
- 每章节保存最近 20 条快照（可配置），支持按时间点恢复。
- 离开页面保护：有未保存改动时弹确认框，防止误丢稿。

## 10. 多模型管理与选择规范（Dify 动态路由）

你当前后台接入 Dify，这个方案完全可行，建议采用“管理员维护模型配置 + 前台按场景选择”。

### 10.1 模型配置来源（管理员维护）

- 保留 `ai_prompts` 负责动作提示词。
- 新增 `ai_models`（建议）负责模型路由，不把所有模型信息塞入 `ai_prompts`。

建议表结构：`ai_models`

- `id`: uuid
- `model_key`: string（唯一键，前后端传输用）
- `name`: string（展示名）
- `action_type`: string（可为空；为空表示通用模型）
- `dify_api_key`: string（对应 Dify App Key）

### 10.2 前端模型选择规则

- 编辑页顶栏提供模型下拉，展示：模型名 + 场景标签（如“润色快”“长文强”）。

- 当模型被管理员停用时，前端提示“模型已下线，请重新选择”。

### 10.3 接口契约补充

- `POST /api/generator` 入参新增：`book_id`, `chapter_id`, `model_key`。
- `POST /api/chat` 入参新增：`book_id`, `model_key`。
- 服务端校验 `model_key` 是否 active，再解析出 `dify_api_key` 调 Dify。

## 11. 双角色登录、后台能力与历史持久化

### 11.1 双角色登录分流

- 统一登录入口 `/auth`（不分两个页面）。
- 平台角色仅保留 `user` 与 `admin` 两种，不引入其他业务角色。
- 登录成功后按 `users.role` 分流：
  - `user` -> `/dashboard`
  - `admin` -> `/admin`
- 所有服务端路由必须再次校验 role，不能只依赖前端跳转。

### 11.2 管理员后台能力边界

- 基础能力：`ai_prompts` CRUD（提示词、动作类型、启用状态）。
- 扩展能力：`ai_models` CRUD（模型展示名、Dify Key、启用状态、排序）。
- 修改后立即生效：运行时读库，不做长时间缓存。
- 高危动作（删除模型/停用模型）需二次确认与操作日志。

### 11.3 用户历史与小说内容持久化

- 小说内容：`chapters.content` 持久化 Tiptap JSON，`word_count` 同步维护。
- 聊天历史：建议新增 `chat_messages`：
  - `id`, `book_id`, `conversation_id`, `role`, `content`, `model_key`, `created_at`
- AI 编辑历史：建议新增 `edit_history`：
  - `id`, `user_id`, `book_id`, `chapter_id`, `action_type`, `source_excerpt`, `result_excerpt`, `model_key`, `created_at`
- 账单增强：`billing_logs` 建议新增 `model_key` 字段，用于对账与模型成本分析。

### 11.4 对话框（Chat UI）明确要求

- 支持将回复“插入到光标处”或“追加到章节末尾”。
- 支持会话记忆：以 `books.current_conversation_id` 维持连续上下文。

## 12. 账号级风控与封禁机制（防提示词盗取）

### 12.1 风险检测目标

- 重点防护对象：提示词窃取、密钥套取、越权指令、系统规则泄露。
- 防护接口：`/api/generator`、`/api/chat`、后台 prompt/model 管理接口。
- 风险请求示例：
  - 要求模型输出系统提示词
  - 要求泄露 API Key 或内部配置
  - 要求忽略系统规则并暴露隐藏指令

### 12.2 自动封禁策略

- 用户连续命中高风险盗取规则达到阈值（建议 3 次）即触发自动封禁。
- 自动封禁动作：
  - `users.status` 更新为 `banned`
  - 立即阻断写作、保存、AI 生成、聊天、导出、知识库写入
  - 所有相关 API 统一返回 `ACCOUNT_BANNED`
- 被封禁账户默认不可自行恢复，必须走申诉流程。

### 12.3 申诉与永久封禁

- 被封禁用户可提交截图/录屏申诉材料。
- 管理员审核结论：
  - 通过 -> 解封并重置风险计数
  - 拒绝 -> 维持封禁
- 按运营策略，无法提供有效证明可执行永久封禁。

### 12.4 审计日志（必须）

- 新增 `security_audit_logs`（建议）：
  - `id`, `user_id`, `action`, `risk_level`, `rule_hit`, `request_excerpt`, `ip`, `ua`, `created_at`
- 记录内容：
  - 风险命中与封禁触发
  - 管理员封禁/解封操作
  - 申诉处理结果
- 审计日志仅管理员可见，普通用户无权访问。

## 15. 脑洞功能极简规则（3字段版）

### 15.1 目标

- 在不改现有 Dify 提示词的前提下，用最少输入完成同人脑洞生成。
- 降低用户填写负担，同时保留 OOC 风险控制能力。

### 15.2 固定输入字段（只保留 3 项）

- `IP`（必填）
- `角色`（必填）

约束：

- 禁止新增第 4 个参数字段。

- 表单使用单页一次填完模式。

### 15.3 三模式联网策略（保留）

- 脑洞模式：联网 `ON`
- 正文模式：联网 `OFF`（仅保留“段落查证”手动触发）
- 审核模式：联网 `ON`

说明：

- 正文默认不自动联网，保证写作流畅度。
- 联网失败时允许降级为“本地知识库上下文生成”。

### 15.4 提示词策略（不改）

- 不新增“固定输出结构”提示词。
- 继续使用你当前已配置的 Dify 提示词。
- 本规则仅调整“输入与流程”，不改提示词资产。

### 15.5 结果去向（必须保留）

- `加入知识库`
- `插入正文`

说明：

- 脑洞结果可沉淀为后续正文与审核可复用资产。

### 15.6 页面交互最小要求

- 入口按钮：`脑洞生成`
- 弹窗字段：`IP` / `角色` / `时间线`
- 提交按钮：`开始生成`
- 结果区操作：`加入知识库`、`插入正文`

### 15.7 验收标准

- 用户 30 秒内可完成表单提交。
- 生成流程不要求额外字段。
- 不修改现有 Dify 提示词即可工作。
- 联网策略按三模式生效（脑洞 ON、正文 OFF、审核 ON）。

## 16. 联网查证 V1/V2 迭代策略（主文档统一版）

### 16.1 目标

- 建立“先可用、再增强”的联网查证路线。
- 在 Dify 后端能力边界内，先保证稳定可用，再逐步提高参数可控度。

### 16.2 V1（先上线）

  - 审核：联网 `ON`

- 与第 15 章不冲突：第 15 章定义“最小输入”，本章定义“阶段迭代”。

## 17. 无支付先上线 + 支付后续开发策略

  - 调整对象

  - 原因备注

- 人均消耗字数（可选）

- 保留“充值/会员”入口按钮。

### 17.5 Phase B（后续支付接入）数据预留

- 建议预留表：
  - `plans`（套餐）

  - Phase A 稳定运行 2-4 周
  - 用户充值诉求明确
  - 有足够资源投入支付联调与风控

### 17.7 验收标准

- 无支付情况下核心业务闭环稳定可用。
- 管理员可运营余额，且有完整审计记录。

## 20. 监控与告警策略

### 20.1 监控对象

- API 层：
  - `/api/generator`：成功率、错误率、P95 延迟
  - `/api/chat`：成功率、错误率、P95 延迟
- 上游依赖：
  - Dify 请求超时率
  - Dify 错误响应率
- 业务核心：
  - 自动保存失败率
  - 扣费写库失败率
  - 封禁触发频次异常波动

### 20.2 告警阈值建议

- 5 分钟窗口内 API 错误率 > 5%：触发告警。
- P95 延迟持续超阈值（如 > 8 秒）：触发性能告警。
- 扣费写库失败率 > 1%：触发高优先级告警。
- 自动保存失败率持续异常：触发用户体验告警。

### 20.3 告警分级与响应时限

- 信息级（Info）：记录并观察。
- 警告级（Warning）：30 分钟内响应。
- 严重级（Critical）：立即响应并启动故障处理流程。

### 20.4 监控上线门禁

- 未接入核心 API 与账务指标监控，不得进入生产上线。
- 联调阶段必须验证告警可触发、可接收、可恢复闭环。

## 22. AI 安全代理与国内部署约束（强制）

### 22.1 Dify 调用安全边界

- 所有 Dify API 必须经过 Next.js 服务端代理接口。
- 严禁前端直连 Dify；严禁在前端暴露 API Key 与端点地址。
- Dify 相关环境变量必须使用服务端变量（禁止 `NEXT_PUBLIC_` 前缀）。

### 22.2 生成前审核强约束

- 所有 AI 生成类请求必须先走审核接口。
- 审核不通过时，生成接口必须阻断并返回明确错误码。
- 审核通过后，才允许进入真实生成流程。

### 22.3 代码规范要求

- 新增核心逻辑必须带中文注释，说明意图、边界与异常处理。
- 代码结构需按模块拆分：路由、服务、数据访问层分离。
- 提交前必须通过 lint 与构建检查，确保可直接运行无报错。

### 22.4 国内部署兼容要求

- 不依赖海外无法稳定访问的 CDN 作为关键链路。
- 前端字体与静态资源优先本地化或可替代源。
- 应用与 Dify 尽量同区域部署，减少网络时延与超时风险。
